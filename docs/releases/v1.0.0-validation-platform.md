## Overview

Validation Platform v1.0.0 completes the deterministic AI Response Validation
Platform. The platform takes a raw AI analysis response, normalizes it into a
canonical structure, and evaluates it through a governed, layered set of
validation rules that produce a single deterministic `ValidationResult`.

Every rule is deterministic: given the same input, the platform always produces
the same verdict, the same issues, and the same statistics. No rule relies on
non-governed input, and no rule performs non-deterministic judgement.

This release becomes the foundation for all future AI engineering capabilities.
Subsequent initiatives build on the normalization handoff, the validation
framework, the governed profile set, and the execution-package persistence
established here.

## Major Deliverables

The following work is complete in this release:

- **Response Normalization** — the multi-stage normalization framework and
  response normalizer that recover a canonical `ParsedResponse` from a raw AI
  response and emit a `NormalizationResult`.
- **Validation Framework** — the rule contract (`ValidationRule`), rule metadata,
  rule layers, and validation exceptions.
- **Validation Registry** — the ordered catalogue that owns rule registration
  and layer ordering.
- **Validation Pipeline** — the deterministic execution engine that runs the
  registered rules in governed layer order.
- **Response Validator** — the component that consumes a `ValidationInput` and
  returns a complete `ValidationResult`.
- **Validator Factory** — the composition root (`build_response_validator`,
  `build_response_validator_for_profile`) that assembles the registry, pipeline,
  and validator.
- **Validation Profiles** — the six governed, immutable profile definitions and
  their read-only registry.
- **PlatformContext Integration** — `PlatformContext` exposes validator
  construction and profile selection as pure dependency composition.
- **CLI Integration** — the `analyze` command exposes `--validate` and
  `--validation-profile` and runs the optional Response Validation phase.
- **Validation Persistence** — the execution writer persists the canonical
  `validation_result.json`.
- **Validation Reporting** — the `ValidationReportBuilder` renders the
  human-readable `validation_report.md`.
- **Governance Synchronization** — the ADR set, capability matrix, coverage
  dashboard, and architecture freeze index are aligned with the implemented
  platform (CAP-044A).
- **Repository Production Readiness Audit** — the repository quality audit and
  production-readiness verification (CAP-044).

## Validation Profiles

The platform ships six governed validation profiles. Each profile is a permanent
governed identity; a profile selects which layers' rules run and never changes
rule ordering. Profiles are owned solely by the `ValidationProfileRegistry`.

| Profile | Enabled layers |
| --- | --- |
| `default` | Transport, Syntax, Schema, Content, Reasoning |
| `strict` | Transport, Syntax, Schema, Content, Reasoning |
| `transport-only` | Transport |
| `syntax-only` | Transport, Syntax |
| `schema-only` | Transport, Syntax, Schema |
| `content-review` | Transport, Syntax, Schema, Content, Reasoning |

`default` is the profile applied when none is explicitly requested. `default`,
`strict`, and `content-review` intentionally enable the same layers today but
remain distinct governed identities so any of them may diverge later without an
architectural change.

## Execution Package

When validation runs and a `ValidationResult` is present, the execution writer
appends two validation artifacts to the execution package:

- **`validation_result.json`** — the canonical persisted form of the
  `ValidationResult`, produced by serializing the model (`by_alias`). This is the
  machine-readable record of the run's verdict, issues, summary, and statistics.
- **`validation_report.md`** — the human-readable twin of the result, rendered by
  the `ValidationReportBuilder` from the same execution data.

**Manifest integration** — both validation artifacts are emitted as part of the
package's generated artifacts, so they appear in the manifest's generated-artifact
listing alongside the other execution outputs. The two files are appended only
when a validation result exists; when validation is not run, the package is
unchanged.
